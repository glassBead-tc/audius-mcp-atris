# 04 — Instructions & Learning Surface

*Domains D-E (instruction fidelity) and D-F (learning surface). After alignment, the
server's self-description is true and stays true, and the agent can learn the API by
reading instead of brute-forcing it.*

A minimal **call surface** (few tools) must not mean a minimal **information surface**.
Resources and prompts cost zero tools and zero context until used — they are the right
home for everything an agent needs to *learn*.

## Truthful instructions (AX-17) · `src/mcp/McpServer.ts` · R1

The `INSTRUCTIONS` string returned on `initialize` is corrected:

- It says **five tools** (it said "two"), and names each with its real role.
- The phantom workflow names ("Trending Snapshot", "Track Genealogy" — which never existed
  in the resource) are gone; the workflow section either lists the 13 real recipes or
  defers to the resource.
- It **names `audius://workflows`, `audius://api/types`, and `audius://sandbox/runtime`
  explicitly** and tells the agent to read them.
- The recommended flow is the real one: *discover* (`search`) → *understand*
  (`inspect_endpoint`) → *execute* (with projection) — not the old "use search first"
  that walked agents straight into the payload bomb.
- It teaches, in one or two sentences each: the `{ data: [...] }` response envelope; the
  projection idiom (return conclusions, not raw rows — the sandbox is a *reducer*); and
  that API errors come back as typed `{ok:false, …}` directives.

## Documentation as a projection of the registries (AX-18) · `src/mcp/McpServer.ts` · R3

`INSTRUCTIONS` is no longer a single hand-maintained constant decoupled from what it
describes. Its volatile sections are **derived**:

- The tool-list section is generated by mapping over the actual tool-definitions array.
- The workflow section is generated by mapping over the workflow titles in
  `resources/Workflows.ts`.

A startup assertion verifies that every workflow name mentioned in prose exists in
`WORKFLOWS_CONTENT`. After alignment, "two vs five tools" and "phantom workflow names" are
not just fixed — they are **unrepresentable**, because the text is a function of the data.

## Exposed type declarations (AX-19) · `src/mcp/McpServer.ts`, `src/sandbox/TypeGenerator.ts` · R2

`TypeGenerator` already produces a ~26 KB `declare const audius` namespace covering all
219 endpoints. Before alignment it was injected into the sandbox as the global
`__apiDecls__` and **never documented** — an unreachable, unused reference artifact.

After alignment the same artifact is surfaced two ways:

- As the MCP resource **`audius://api/types`** — readable by any agent, cacheable client-side.
- As a documented in-sandbox global, **`audius.types`** (a string), referenced by the
  `audius://sandbox/runtime` manifest.

No new generation work — the artifact already exists; alignment only stops hiding it.

## Workflows as MCP prompts (AX-20) · `src/mcp/Prompts.ts` *(new)*, `src/resources/Workflows.ts` · R2

The 13 workflow recipes in `Workflows.ts` are the server's single most valuable teaching
asset and its proof of the "good path". After alignment they are **triple-surfaced** from
one source of truth, for three ways an agent discovers things:

1. **As MCP prompts** (`src/mcp/Prompts.ts`) — each workflow is a parameterized prompt
   (`audius:rising-stars`, `audius:genre-report`, `audius:artist-compare`, `audius:bpm-landscape`,
   …) with args like `genre` / `artist` / `limit`, returning a projection-correct
   `execute` body. An agent that knows the *task* gets vetted code in one step.
2. **As the resource `audius://workflows`** — for the agent browsing capabilities (unchanged
   content).
3. **As inline examples** inside `inspect_endpoint` output — for the agent that found an
   endpoint and wants to see it used in context.

`prompts/list` returns the 13 prompts (it returned `[]` before); the `prompts` capability
is declared in `initialize`.

## Documented implicit contracts (AX-21) · `src/sandbox/TypeGenerator.ts`, `McpServer.ts` · R3

Two contracts an agent previously learned only by trial and error are now stated:

- **The `{ data: [...] }` envelope** — declared in the `TypeGenerator` header comment and
  in `INSTRUCTIONS`. One sentence removes a whole class of `.data` confusion.
- **Per-endpoint auth needs** — `inspect_endpoint` emits `requiresAuth`; the generated
  type declarations mark user-scoped endpoints (e.g. `/me*`) with a 🔒 token. Agents learn
  auth needs before a 401, not after.

## Resource & prompt surface, after alignment

| URI / name | Kind | Content |
|---|---|---|
| `audius://workflows` | resource | The 13 analysis recipes (markdown) |
| `audius://api/types` | resource | Generated TypeScript declarations for all 219 endpoints |
| `audius://sandbox/runtime` | resource | The sandbox capability manifest (see [`03`](./03-sandbox-runtime.md)) |
| `audius:rising-stars`, `audius:genre-report`, … (13) | prompts | Parameterized, projection-correct `execute` bodies |

## Definition of done

- [ ] `INSTRUCTIONS` states five tools, names the resource URIs, teaches projection + the
      `{data}` envelope, and contains no phantom workflow names.
- [ ] The tool-list and workflow sections of `INSTRUCTIONS` are derived, not hand-written;
      a startup assertion guards workflow-name fidelity.
- [ ] `audius://api/types` and `audius://sandbox/runtime` resources exist.
- [ ] `prompts/list` returns the 13 workflow prompts; the `prompts` capability is declared.
- [ ] `inspect_endpoint` carries `requiresAuth`; type declarations document the `{data}` envelope.
